# LLM 综合判断接入契约

日期：2026-05-08

## 目标

本系统后续可以接入其他 LLM，用于在规则基线、项目事实和证据材料基础上生成模型主导的最终研判、复核建议和方案深化意见。规则评分不再是最终裁判，而是模型研判的基线、参照物和差异检查坐标。

## 输入边界

LLM 应只接收后端 `/api/evaluate` 或导出 JSON 中的结构化结果，核心输入为：

- `project`：项目、站点、地块和附件输入。
- `score` / `scorePercent` / `level`：后端规则计算出的分值和等级。
- `dimensions`：四维评分明细，含因子、选项、权重、加权分、来源和是否临时试算。
- `recommendation`：推荐方案、备选方案和命中规则。
- `missing`：正式评分字段和项目资料缺项。
- `stationContext`：客流、出入口接口、运营配套和周边配套。
- `llmReviewContext`：面向 LLM 的事实、引用、维度拆解、方案比选、风险复核和输出约束。
- `researchPlan`：系统自动生成的研究问题、动态维度、标杆案例和检索任务。
- `evidencePack`：本地知识库、五类标杆案例缓存、独立搜索和模型联网证据。
- `modelAssessment`：模型或本地兜底生成的动态维度研判、风险、不确定性和复核问题。
- `modelJudgement`：模型最终等级、推荐方式、置信度、判断理由、风险和补资优先级。
- `modelRuleDifference`：规则基线与模型结论的一致或覆盖关系、覆盖理由和人工复核标签。
- `modelQuality`：事实诚实性、依据覆盖、缺项影响和输出结构状态。
- `diagramBrief`：面向示意图 SVG 预览和报告嵌入的结构化图示 brief。
- `reportModes`：客户正式版、专家附录版、领导汇报版三类报告模式。
- `capabilityStatus`：本地缓存、独立搜索、LLM、模型联网的可用状态和降级原因。

## 规则基线与模型主导

LLM 是一期最终研判中枢，可以在明确说明理由、证据、置信度和复核标签的前提下覆盖规则评分对应的等级或推荐方案。后端规则评分不再是最终裁判，而是模型研判的基线、参照物和差异检查坐标。

系统必须同时保留：

- `ruleBaseline`：规则分数、规则等级、规则推荐方式、维度拆解和命中规则。
- `modelJudgement`：模型最终等级、推荐方式、置信度、判断理由、风险和补资优先级。
- `modelRuleDifference`：规则结论与模型结论的一致或覆盖关系、覆盖理由和人工复核标签。
- `modelQuality`：事实诚实性、依据覆盖、缺项影响和输出结构状态。

模型不能把缺失事实写成已确认事实。接口标高、消防分区、产权界面、管线条件、施工图审查和工程级图纸结论若未被输入资料确认，必须标注为“模型推断”或“需人工复核”。

## 引用与证据要求

每段综合判断应尽量引用 `llmReviewContext.referenceBasis` 中的资料来源。当前可引用的主要来源包括：

- 《互联互通智能体研发工作方案0318.pptx》。
- 《评价因子赋值明细表.xlsx》。
- 20260507 反馈口径。
- 《车站TOD级别划分整理.xlsx》。
- 《每站每月日均进站.xlsx》。
- 《苏州地铁运营站点出入口、联通接口情况梳理统计表20260128.xlsx》。
- 《运营公司车站出入口开放及周边配套统计表202604.xlsx》。
- 设计细则、设计指引和知识库规则卡。

站点周边配套、出入口、联通形式可以作为预评估线索，但不能替代具体地块的正式用地性质、开发强度、地下空间功能和接口标高。

## 推荐输出结构

建议外部 LLM 按以下结构输出：

1. 综合判断结论。
2. 主要依据与引用来源。
3. 推荐方案适用性说明。
4. 备选方案触发条件。
5. 风险与需补齐资料。
6. 近期 / 中期 / 远期实施建议。
7. 人工复核问题清单。

## 风险控制

- 对缺资料字段必须保留“预评估”或“待补齐后复核”表述。
- 对工程可行性、消防、结构、管线、权属、接口标高等内容，只能提出复核建议，不能直接判定已满足。
- 对引用来源不充分的判断，应标注为推断。
- 对模型输出应保留原始后端 JSON 和模型响应，便于追溯。
- 模型和联网能力均为研判层核心能力；未配置或调用失败时，系统必须提示用户确认，确认后才使用本地知识库和五类标杆案例缓存生成离线兜底研判。
- 实时外搜优先于缓存补强，但不得阻断规则评分、报告导出和资料补齐清单。
- 未被本地资料或独立搜索印证的模型联网信息，只能作为低置信复核线索。
- 模型可用时以模型综合研判为主，本地资料、标杆案例和搜索结果作为模型判断的证据弹药，不反客为主。

## 分层增强策略

系统按“规则基线、实时研究、本地知识兜底、模型主导”执行：

1. 规则引擎输出评分基线、推荐基线和维度拆解。
2. 研究层生成研究问题和动态研判维度。
3. 证据层先尝试独立搜索和模型联网，并准备本地知识库与五类标杆案例缓存作为证据弹药。
4. 模型层生成最终综合研判；模型可覆盖规则等级或推荐方案，但必须保留规则基线、覆盖理由、证据引用和人工复核标签。
5. 报告层必须展示能力状态、证据模式、置信度、差异检查和人工复核问题。

第一批标杆案例为：金家堰邻里中心、清树湾站东南象限青年公寓、乐桥站、阳澄湖中路站、高铁苏州北站。

## 验收

报告丰富度和 LLM 接入框架由以下脚本校验：

```powershell
$env:NODE_PATH='C:\Users\R\.cache\codex-runtimes\codex-primary-runtime\dependencies\node\node_modules'
& 'C:\Users\R\.cache\codex-runtimes\codex-primary-runtime\dependencies\node\bin\node.exe' .\tools\verify_report_richness.cjs
```

该脚本会检查报告章节数、引用依据数量、四维拆解、方案比选、风险复核点和 LLM 输出结构约束。

模型导向研究链路由以下脚本校验：

```powershell
python .\tools\verify_model_oriented_research.py
```

该脚本会检查五类标杆案例是否均能生成 `researchPlan`、`evidencePack`、`modelAssessment` 和 `capabilityStatus`，并确认离线模式下可使用本地兜底研判。
