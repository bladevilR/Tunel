## 结论

当前 `outputs/amap_js_3d` 这一版不能作为后续交付基础继续修。它证明了“高德 JS 3D 底图 + 可控覆盖层”这个技术路径可以跑通，但当前画面和用户提供的人工底图不一致，尤其是红线、河道、站点出口、通道之间的空间关系没有校准好。

下一工作分支建议从“人工底图/确定性标注数据”重新建模，不要沿用当前 hardcoded 几何。

## 用户目标

用户需要的是类似人工示例图的通道示意图：

- 底图来自高德/百度地图或人工截图。
- 红线应贴合地块边界，尤其要体现沿河边界关系。
- 地铁站体、出入口、拟建通道、地块之间的位置关系要直观。
- 成果不是施工图，不要求严肃工程数值精度，但不能出现明显空间逻辑错误。
- 客户想要 3D 效果图，但不能接受 AI 生图的随机性。

## 当前实现方式

当前 demo 使用：

- 高德 JS API 2.0 加载地图底图。
- 地图中心曾校正到群力站片区附近。
- 红线、通道、拟建体块、站体是 HTML Canvas 覆盖层绘制。
- `AMap.Object3DLayer` / `AMap.Object3D.Prism` 在当前运行环境没有暴露，因此没有使用高德自身的自定义 3D 体块能力。
- 当前自定义 3D 建筑和通道是 Canvas 模拟体块，不是真实 GIS/BIM 模型。

## 当前失败点

1. 红线几何不是从用户人工图或真实地块数据提取的，而是手工示意点位。
2. 红线和河道关系不对，用户指出原图红线沿河，而当前图因为缩放/视角/几何综合问题，看起来把河道关系盖住了。
3. 通道、站体、出入口、地块红线之间只是示意摆放，没有按人工底图逐点校准。
4. 视角比例多次调整仍不能解决根本问题，因为底层几何不是可靠来源。
5. 高德 3D 底图可以作为背景，但不能替代工程侧矢量标注数据。

## 已验证的技术事实

- 高德 JS API 页面可以本地加载。
- 页面可通过本地服务注入 API Key，不必把 Key 写进 HTML。
- 当前验收脚本能检查页面是否加载、地图对象是否存在、覆盖层是否渲染、是否有页面错误。
- 高德 3D 底图可用于浏览器展示和截图。
- 当前环境没有拿到可用的高德自定义 3D Prism 接口，拟建体块需要自绘或交给 Blender/Three.js。

## 不建议继续做的事

- 不要继续在当前 hardcoded 红线基础上微调。
- 不要继续靠 zoom、pitch、center 去掩盖几何不准。
- 不要用 image2 / nano banana 直接从地图截图生成最终图，随机性不可控。
- 不要把 AI 生成图作为主成果，只能作为后期风格化参考。

## 下一分支建议

新分支应改成“先有确定底图和确定标注，再做 3D 风格化”的流程：

1. 以用户提供的人工高德截图作为第一版底图来源。
2. 人工或半自动提取红线、站体、出口、通道锚点，保存成 JSON/GeoJSON。
3. 用固定渲染程序生成示意图，不让模型随机改空间关系。
4. 3D 效果优先用 Canvas/Three.js/Blender 确定性渲染。
5. 若要接高德 API，优先使用它提供底图和定位，不依赖它生成拟建建筑。

## 新分支最小可行目标

先做一个“人工底图 + 固定标注 + 轻 3D 体块”的版本：

- 输入：用户提供的 PNG 底图。
- 输入：红线点、站体点、出口点、通道线。
- 输出：一张 PNG 示意图。
- 要求：红线不遮挡河道关系，通道连接站口和地块，建筑体块在红线内，整体效果接近用户提供样图。
